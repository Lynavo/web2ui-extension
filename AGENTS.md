# AGENTS.md

This repository contains the standalone, local-first Web2UI Chrome/Chromium extension. These
instructions apply to the entire repository and are intended for coding agents and automated
contributors.

## Product boundary

The public edition has one workflow:

1. The user explicitly captures the visible area or full page.
2. The extension reads and converts that page locally.
3. The user explicitly copies the result and pastes it into Figma.

Dynamic content uses the documented `local-static-v1` profile: static DOM/CSS/text conversion
remains correct, while Canvas, WebGL, video, animation, and browser-only effects receive at most
one current-frame sample per attempted fallback region, with explicit warnings. Do not add hosted
recovery, multi-frame quality retries, animation convergence, or multi-shot alpha recovery to the
public edition.

Keep that boundary intact. Do not add accounts, billing, hosted APIs, uploads, telemetry, remote
configuration, remote executable code, a Figma plugin runtime, or Chrome Web Store publishing.
Network access is limited to fetching HTTP(S) assets already declared by the page being captured.

The repository is an independent AGPL-3.0-only project. Do not copy private code, fixtures,
credentials, operational details, or internal documentation from another edition or repository.

## Architecture invariants

- `src/core/capture` measures browser facts and produces capture data. Browser-only facts must be
  measured here, not guessed later from names, URLs, or content shape.
- `src/core/contracts` defines untrusted-data interfaces. Validate data when it crosses a runtime
  seam and preserve backward-compatible optional fields when extending a contract.
- `src/core/conversion` is deterministic and side-effect free. It must not use Chrome APIs,
  IndexedDB, the DOM, filesystem access, or network access.
- `src/extension` adapts Chrome MV3 interfaces to the core modules. Keep Chrome-specific behavior
  out of `src/core`.
- Content messages are accepted only for the active run, tab, and document identity. Never weaken
  this three-part check.
- Asset reads omit credentials, accept only HTTP(S), and remain bounded by byte and time limits.
- The extension stores at most one current RenderPlan. Retention, size limits, explicit clearing,
  and expiry alarms must remain aligned across code, tests, README, and PRIVACY.
- All executable release code is packaged locally. Do not introduce `eval`, dynamic remote
  imports, remote scripts, source maps, secrets, or machine-local paths into `dist/`.

## Development workflow

Use Node.js 24 or newer and pnpm 10.

```bash
pnpm install --frozen-lockfile
pnpm validate
```

During development, run the smallest relevant test first. Before handing off a code, manifest,
build, dependency, or release change, run `pnpm validate`. Before a release-related handoff, also
run `pnpm package` and `pnpm audit`.

Add or update tests with every behavior change:

- capture/extraction behavior: `tests/core-capture-*.test.ts` or content tests;
- pure conversion behavior: `tests/core-conversion*.test.ts`;
- Chrome runtime and message behavior: background/content/state tests;
- manifest, repository, build, package, and release policy: `*.test.mjs`;
- complete user flow: `tests/e2e/local-copy-flow.test.ts`.

For fidelity defects, add a minimal synthetic fixture or focused contract test before changing
the implementation. Never commit captures from private or authenticated pages.

## Repository hygiene

- Generated artifacts belong only under `dist/` or `out/`; both are ignored.
- Do not commit empty placeholder directories, browser profiles, screenshots generated by tests,
  ZIP/CRX packages, keys, environment files, or dependency directories.
- Keep public documentation concise and factual. Do not add internal plans, agent transcripts,
  run logs, store-listing material, or speculative roadmaps.
- Permission changes require matching updates to `README.md`, `PRIVACY.md`, tests, and the release
  boundary verifier.
- User-visible changes require `CHANGELOG.md`. Architecture or workflow changes require the
  matching document under `docs/`.
- Use conventional commit subjects. Keep pull requests focused and explain security, privacy, and
  permission impact.

## Definition of done

A change is complete when its relevant tests pass, `pnpm validate` is green when required,
generated artifacts remain untracked, public documentation matches behavior, and no product,
privacy, or release boundary has been weakened.
